# Files & Sidecars

MIVF uses sidecar files placed next to your `.mivf` video file for additional features.
A growing subset of these can alternatively be embedded directly into the `.mivf` ‚Äî see
[MIVF Asset Bundle](#mivf-asset-bundle-masb-embedded-assets) below.

## Supported Sidecar Files

Place any of these in the same folder as `yourvideo.mivf`.

### Subtitles

| File | Purpose |
| :--- | :--- |
| `yourvideo.srt` | Main subtitle track |
| `yourvideo.1.srt`, `yourvideo.2.srt`, ‚Ä¶ | Additional subtitle tracks |

Cycle through tracks during playback with **Y**. Subtitle delay and on-screen position
can be adjusted in Settings. A cue with more than one line now renders as two real,
separately centered lines (previously they were joined with a `" | "` separator); a
third or later line still folds into the second line, since the renderer only supports
two lines. See [Subtitles (player)](../player/subtitles.md) and
[Creating Subtitles](subtitles.md).

### Chapters

| File | Purpose |
| :--- | :--- |
| `yourvideo.chapters` | Chapter markers |

Format: one entry per line.

```
# comment lines start with #
90 Opening
05:30 Chapter Name
1:12:45 Credits
```

- Plain seconds (`90`)
- `M:SS` or `H:MM:SS`
- Optional label after the timestamp

Navigate chapters during playback with **R + ‚Üê/‚Üí**. If a `.chapthumbs` file (below) is
also present and matches this file's chapter count, the DVD-style menu's Scene Selection
page shows thumbnails instead of a plain list. See [Chapters (authoring)](chapters.md).

### Cover Poster and Preview

| File | Purpose | Format |
| :--- | :--- | :--- |
| `yourvideo.cover` | Browser poster (list-view size) | Raw RGB565, headerless, exactly 88√ó50 |
| `yourvideo.preview.cover` | Larger browser preview | Raw RGB565, headerless, exactly 176√ó100 |

When the Library shows a preview, it resolves artwork in this order:

1. **Already-displayed match** ‚Äî if the in-memory 176√ó100 buffer already holds this
   exact file's artwork (nothing changed since it was last shown), reuse it as-is.
2. **4-slot RAM cache** ‚Äî a small least-recently-used cache of 176√ó100 images kept for
   the current browsing session, checked by file path.
3. **`.preview.cover` sidecar**, if present ‚Äî loaded straight into the 176√ó100 buffer
   and added to the cache above.
4. **`.cover` sidecar**, if present ‚Äî loaded at its native 88√ó50, then the player
   builds a **softened 176√ó100 in-memory display copy** (box-filter averaging, not a
   blocky nearest-neighbor stretch) and adds *that* to the cache. This generated copy
   is session-only RAM ‚Äî the player never writes a `.preview.cover` file back to the
   SD card.
5. **First successfully decoded preview frame** from the video itself, if neither
   sidecar is present.
6. If none of the above produce artwork, the Library shows a generic placeholder card.

`.preview.cover` itself is an optional, **separately authored** 176√ó100 RGB565 sidecar ‚Äî
if you're authoring one from a source image, any 176√ó100 RGB565 image works; it does not
need to be derived from an 88√ó50 `.cover`. The box-filter upscale described in step 4 is
something the player does *for you* only when no `.preview.cover` exists, purely to
avoid a blocky stretch of the smaller legacy image ‚Äî it's not how `.preview.cover` files
are meant to be produced. See [Artwork](artwork.md) for how to prepare these.

### Synopsis

| File | Purpose |
| :--- | :--- |
| `yourvideo.nfo` | Short description shown in the browser preview panel |

Plain text. Displayed capped at roughly 19 characters per line, two lines ‚Äî longer text
is truncated, not wrapped further.

### Seek Index

| File | Purpose |
| :--- | :--- |
| `yourvideo.idx` | Seek index sidecar for fast seeking |

Auto-generated by the encoder. The encoder also embeds a seek index footer directly in
the `.mivf` file. See [Seek Index](../technical/seek-index.md).

### DVD-Style Menu

| File | Purpose | Format |
| :--- | :--- | :--- |
| `yourvideo.menu.ini` | Menu definition: title + up to 8 buttons (`play`/`resume`/`chapters`/`back`) | Plain INI |
| `yourvideo.menu_bg.cover` | Animated menu background | Raw RGB565, headerless, exactly 400√ó240 |
| `yourvideo.chapthumbs` | Scene Selection thumbnails, one per chapter | Binary: `MCTH` magic + version + count + 96√ó54 dimensions + `count` RGB565LE frames |
| `yourvideo.screensaver.cover` | Idle-screensaver image (root menu view only) | Raw RGB565, headerless, exactly 96√ó54 (10,368 bytes) |

See [Menu Authoring](menu-authoring.md) for exactly how to build each of these, and
[Movie Menus](../player/movie-menus.md) for how the player uses them at runtime. If
`yourvideo` has no `.menu.ini`, it plays directly with no menu, exactly like before.

## MIVF Asset Bundle (MASB, embedded assets)

`tools/mivf_embed_assets.py` can pack sidecar files directly into a `.mivf` as an
appended, self-contained bundle instead of shipping them as separate files ‚Äî useful for
distributing a single file per movie. **Currently only `menu_bg.cover` is actually read
back by the player** (`mivf_menu_load_background`); `menu.ini`, `.chapters`, `.nfo`, and
`.idx` can be packed into the bundle for forward compatibility, but the player still
reads those specific four from their sidecar files today, not from the bundle. A sidecar
always wins over an embedded copy of the same asset when both are present. Chapter
thumbnails (`.chapthumbs`) and the screensaver image currently have **no** embedded-asset
support at all ‚Äî they are sidecar-only.

Inspect what's embedded in a file with:

```bash
python tools/mivf_list_assets.py yourvideo.mivf
```

## App Data Layout

Settings, bookmarks, and other persistent data live under:

```
sdmc:/3ds/mivf_player_3ds/appdata/
‚îú‚îÄ‚îÄ settings.ini          Player settings (including persisted volume, 0-300%)
‚îú‚îÄ‚îÄ favorites.ini         Favorite video paths (capped at 128 entries)
‚îú‚îÄ‚îÄ recents.ini           Recently opened videos (capped at 16, most-recent order)
‚îú‚îÄ‚îÄ bookmarks/            Per-video resume positions
‚îú‚îÄ‚îÄ logs/                 Debug logs
‚îú‚îÄ‚îÄ cache/                Cached data
‚îî‚îÄ‚îÄ benchmarks/           Benchmark data
```

Note: the DVD-style menu's "last highlighted button/chapter" memory is **not** part of
this appdata layout ‚Äî it's a plain in-memory value, scoped to the current app session
only, and is never written to any of the files above.

Legacy root-level files (`sdmc:/mivf_settings.ini`, `sdmc:/mivf_favorites`,
`sdmc:/mivf_recents`, `sdmc:/mivf_bookmarks/`) are still read for migration but new data
is always written to the appdata folder.
